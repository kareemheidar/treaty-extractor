Your build guide

Treaty Extractor — build it in Retool, step by step

A friendly, click-by-click guide to building your hackathon app. No coding or AI background needed — every step tells you exactly what to click in Retool and what it does. Work through it in order and you'll have a working app by the end.

For: the build team (3–5 people) No experience required You'll use AI, made simple

What are we building?

Right now, after a reinsurance treaty (a contract) is signed, someone reads the whole document by hand and re-types the important details into other systems. It's slow and easy to get wrong. Your app will do the reading for them: upload the treaty → the app pulls out the key details and the rate tables → a person checks them on screen → clicks Approve → the clean data is saved and can be exported.

The goal

One working slice you can show off — upload a treaty, watch it fill in, approve it. Not a finished product.

How to use this

Follow the parts in order. Each step has a Goal, the exact clicks, and a Checkpoint to confirm it worked.

Golden rule

Use practice (mock) data only — never a real client's treaty. And keep everything in English letters/numbers.

Two promises to make this easier

You don't need to memorise anything — copy the snippets we give you and move on. And you won't get stuck alone: your coach stays with you the whole time and there's support on standby. If a step doesn't behave, wave them over.

Learn the words once

1 · Retool basics (5-minute primer)

Retool is a tool for building simple apps by dragging pieces onto a screen — no coding from scratch. Here are the only words you need, in plain English. You'll meet each one again when you use it.

App

The thing you're building — a screen with buttons, tables, and boxes. You build it in the editor.

Component

A ready-made piece you drag onto the screen: a button, a table, a text box, a file upload. You pick them from the component list on the right.

Query

An action the app runs — "read the file", "ask the AI", "save to the database". They live in the panel at the bottom of the editor.

Resource

A connection the app uses — the AI service and the database are each a resource. Your admin has set these up already.

Retool Database

A built-in place to store your data in tables (like a spreadsheet with rules). This is where approved treaties get saved.

AI action

A special query that asks an AI model to do something — read a document, or pull out fields. You just fill in a box; no AI knowledge needed.

Transformer

A tiny helper that tidies up data from a query so it shows nicely in a table. You'll copy ours in.

Event handler

A rule that says "when this happens, do that." Example: "when the AI finishes, save the result." You add these in a few clicks.

The editor at a glance

When you open an app you'll see: the canvas (your screen) in the middle, the component list on the right, the query panel at the bottom, and the page/component tree + inspector on the left/right. That's it — you'll use these four areas the whole time.

Part A · owner: App & Screens + AI & Extraction

4 · Part A — Upload & read the treaty

Goal for this part: you can upload a treaty PDF and see its text on screen. Two short steps.

Step A1 — Create the app & add a file uploadApp & Screens
Goal: an app where you can pick a treaty file.
  1. In your Retool Space, click Create new → App. Name it Treaty Extractor. You're now in the editor.
  2. From the component list (right side), drag a File Button onto the canvas.
  3. With it selected, look at the Inspector (right). Rename it to fileInput at the top.
  4. Drag a Text component underneath. In its Value box, paste: {{ fileInput.value.length ? fileInput.files[0].name : "No file yet" }}

What this does: the File Button lets a user choose a treaty; the Text shows the file's name so you know it worked.

Checkpoint: click your File Button, pick the practice treaty PDF — the file name appears in the Text box.
Step A2 — Turn the PDF into text (first AI query)AI & Extraction
Goal: get the words out of the PDF so the app can use them.
  1. In the query panel (bottom), click + New → Resource query.
  2. For Resource, choose your AI resource (your admin named it — it runs on AWS Bedrock for this event).
  3. Set Action to Convert document to text.
  4. In File source, paste: {{ fileInput.value[0] }}
  5. Rename the query (top-left of the panel) to convertDoc, then click Run.
  6. Drag a Text Area component onto the canvas and set its Value to {{ convertDoc.data }} so you can see the text.

What this does: reads the PDF and gives back its text. This step happens inside Retool — nothing is sent to an AI model yet.

Checkpoint: after Run, the treaty's text appears in the Text Area.

If "Convert document to text" isn't in the list, that feature may not be switched on for the event. No problem — use Plan B below instead. Everything after this step works exactly the same.

Step A2 (Plan B) — Read the PDF with a small script in a WorkflowAI & Extraction
Goal: the same result as A2 — the treaty's text — but using a tiny script instead of the built-in AI action. Only do this if A2 wasn't available.

What's a Workflow? It's a separate mini-program in Retool that runs a few steps in order, away from your app screen. We'll make one whose only job is: take the PDF, read its text, hand it back. Your app calls it like any other action.

  1. In Retool, go to Create new → Workflow. Name it extractText.
  2. It starts with a Start trigger block. This is where your app will send the PDF.
  3. Add a Code block and set its language to Python. Paste the script below into it.
  4. Open the Workflow's settings → Python libraries and add pypdf (this is the tool that reads PDFs). If it's already there, skip this.
  5. Back in your app: query panel → + New → Resource query → choose your extractText Workflow. In its input, set file to {{ fileInput.value[0] }}. Name this query extractText and Run.
  6. Show the result: a Text Area with Value {{ extractText.data.text }}.

Paste this into the Python code block

# Reads the words out of the uploaded PDF
import base64, io
from pypdf import PdfReader

# 1. The app sends the PDF as base64 (a text-safe way to send a file).
raw = startTrigger.data["file"]

# 2. Turn that text back into the real PDF file.
pdf_bytes = base64.b64decode(raw)

# 3. Open the PDF and read the text from every page.
reader = PdfReader(io.BytesIO(pdf_bytes))
pages = [(p.extract_text() or "") for p in reader.pages]

# 4. Send all the text back to the app.
return {"text": "\n\n".join(pages)}

What each part does, in plain words:

  • import… — brings in pypdf, the free tool that knows how to read PDFs.
  • Step 1 — grabs the file your app sent in.
  • Step 2 — files travel as scrambled text (called base64); this un-scrambles it back into a real PDF.
  • Step 3 — opens the PDF and pulls the words off each page.
  • Step 4 — joins the pages together and hands the text back to your app.

One thing to remember: later steps in this guide say to use {{ convertDoc.data[0] }}. If you used Plan B, use {{ extractText.data.text }} in those spots instead — it's the same treaty text, just from the Workflow.

Checkpoint: the Text Area shows the treaty's text, just like in A2.
Part B · owner: AI & Extraction

5 · Part B — Extract the details with AI

Goal for this part: the treaty's fields and its rate table show up as neat tables. This is the AI heart of the app — take it slowly, copy our snippets, and test after each step.

Step B1 — Ask the AI for the treaty fieldsAI & Extraction
Goal: turn the messy text into labelled fields (treaty name, dates, percentages…).
  1. Query panel → + New → Resource query → Resource: your AI resource → Action: Generate text.
  2. For Model, pick the Claude model your admin enabled (for example claude-sonnet-4-6).
  3. In the System message box, paste the instructions below (it tells the AI exactly what to return).
  4. In the Prompt box, paste the prompt below.
  5. Rename the query to extractFields and click Run.

Paste into "System message"

You are a reinsurance treaty analyst. Read the treaty text and return
ONLY a raw JSON object with these fields: treaty_name, cedent_name,
effective_date, termination_date, treaty_type, quota_share_pct,
retention_amount, currency, commission_pct.
For each field also give a short "source_quote" copied from the text
and a "confidence" of high, medium, or low.
If a field is not in the text, use null. Do NOT use markdown or code fences.

Paste into "Prompt"

Extract the treaty fields from the text below.

TREATY TEXT:
{{ convertDoc.data[0] }}

What this does: the AI reads the text and hands back the details as data your app can display.

Checkpoint: after Run, the query's Output area (bottom) shows the fields with values.
Step B2 — Tidy the AI's answer (a transformer)AI & Extraction
Goal: make sure the AI's answer is always clean, even if it adds extra formatting.
  1. Query panel → + New → Transformer. Rename it extractClean.
  2. Delete the sample code and paste the snippet below.
// cleans the AI answer so it always works
let raw = extractFields.data;
if (raw && typeof raw === 'object') return raw;
let s = String(raw).trim()
        .replace(/^```[a-z]*\s*/i, '')
        .replace(/\s*```$/, '').trim();
try { return JSON.parse(s); }
catch (e) { return { error: e.message }; }

Why this exists: sometimes the AI wraps its answer in extra symbols. This helper quietly removes them so your table never breaks. You don't need to understand the code — just paste it.

Checkpoint: the transformer's output shows a clean set of fields.
Step B3 — Show the fields in a tableApp & Screens
Goal: the fields appear on screen in a table people can read.
  1. Drag a Table component onto the canvas.
  2. Set its Data to {{ extractClean.value.fields }} (or {{ [extractClean.value] }} if you kept one row).
  3. Rename the table to paramsTable in the Inspector.
Checkpoint: the treaty's fields show up as rows in the table.
Step B4 — Pull out the rate tableAI & Extraction
Goal: the pricing grid (the rate table) comes out as its own table.
  1. Make another Generate text query (same as B1) named extractRates.
  2. System message: "Extract the rate table as a raw JSON array of rows, each with age_band, male_ns, male_sm, female_ns, female_sm. Copy numbers exactly. No markdown."
  3. Prompt: Extract the rate table.\n\nTREATY TEXT:\n{{ convertDoc.data[0] }}
  4. Add a Transformer named rateTableClean (copy the same cleanup snippet from B2, but change extractFields to extractRates).
  5. Add a Table named rateTable with Data {{ rateTableClean.value }}.
Checkpoint: the rate grid appears as a table of numbers.

Tip: the rate table is the trickiest bit. If it looks off, tell your coach — there's a simpler no-AI way to read a clean grid.

Part C · owner: Data & Database

6 · Part C — Save treaties to the database

Goal for this part: every treaty you analyse gets saved automatically so it shows up in a list you can come back to.

Step C1 — Design & create the storage tableData & Database
Goal: a table (like a spreadsheet with rules) to hold every treaty.

First, decide what to store. A table has columns (the kinds of information) and rows (one per treaty). Here's the design — build exactly this:

id
auto number — the unique key (Retool adds this for you)
treaty_ref
text — the treaty's reference code
cedent
text — the client company
source_file
text — the uploaded file name
status
text — either in_review or approved
extracted_json
json — the AI's original answer (kept as-is)
reviewed_json
json — the human-checked answer
rate_json
json — the rate table
reviewed_by
text — who approved it
reviewed_at
timestamp — when it was approved
created_at
timestamp — when it was first analysed
  1. Go to Resources → Retool Database (left side of Retool, the database icon).
  2. Click Create new → Table. Name it treaties.
  3. Add each column above with its type (text, json, or timestamp). id and created_at are usually added for you.

Why two "json" columns? We keep the AI's first answer (extracted_json) separate from the human-checked one (reviewed_json) so we never lose the original — and later you can see how good the AI was.

Checkpoint: an empty treaties table exists with all the columns.
Step C2 — Build the "save" actionData & Database
Goal: an action that writes one treaty into the table.
  1. Back in the app, query panel → + New → Resource query → Resource: Retool Database.
  2. At the top of the query, switch to GUI mode (the toggle next to SQL). GUI mode lets you fill in boxes instead of writing code.
  3. Action type: Insert a record. Table: treaties.
  4. In the Changeset (the field map), add each value:
    treaty_ref = {{ extractClean.value.treaty_reference }}
    cedent = {{ extractClean.value.cedent_name }}
    source_file = {{ fileInput.files[0].name }}
    status = in_review
    extracted_json = {{ extractClean.value }}
    rate_json = {{ rateTableClean.value }}
  5. Rename the query to insertTreaty.

What this does: saves one treaty as a new row, marked in_review (waiting for a person to check it).

Checkpoint: click Run once — a new row appears in your treaties table.
Step C3 — Make it save automatically (event handler)Data & Database
Goal: the moment the AI finishes, the treaty is saved — no extra click.
  1. Open your extractFields query. In its settings, find Event handlers.
  2. Click + Add. Set Event to Success.
  3. Set Action to Control query → Trigger, and choose insertTreaty.

What this does: an event handler is a "when this, do that" rule. Here: when the AI finishes reading the treaty, save it automatically.

Checkpoint: upload a fresh treaty and run extraction — a new row appears in treaties by itself.
Part D · owner: App & Screens

7 · Part D — The treaty list & review drawer

Goal for this part: a list of every treaty, and clicking one opens a side panel (a "drawer") with two tabs — the fields, and the rate table.

Step D1 — Show all treaties in a listApp & Screens
Goal: a table showing every saved treaty.
  1. Query panel → + New → Resource query → Retool Database → GUI mode → Action: Read / Select → Table: treaties. Name it getTreaties.
  2. In the query's settings, turn on Run on page load so the list is always fresh.
  3. Drag a Table onto the canvas, set Data to {{ getTreaties.data }}, and name it treatiesTable.
Checkpoint: your saved treaties appear in the list.
Step D2 — Add the three status filtersApp & Screens
Goal: buttons to switch between In Review, Approved, and All.
  1. Drag a Segmented Control above the table. Name it statusFilter.
  2. Set its options to In Review, Approved, All. Set the default to In Review.
  3. The simplest filter: on the treatiesTable, use its built-in Filter setting to show rows where status matches the control (treat "All" as no filter). Your coach can show you the exact box.

What this does: lets a reviewer focus on treaties still waiting for them (In Review) by default.

Checkpoint: switching the control changes which treaties show.
Step D3 — Open a drawer when a row is clickedApp & Screens
Goal: click a treaty → a panel slides in from the right.
  1. From the component list, add a Drawer frame. Name it reviewDrawer.
  2. Select treatiesTable → Event handlers → + Add → Event: Row clicked (or "Select row") → Action: Control component → Open → choose reviewDrawer.

What this does: the currently clicked treaty is available as {{ treatiesTable.selectedRow }} — you'll show its details inside the drawer next.

Checkpoint: clicking a row opens the (empty) drawer.
Step D4 — Two tabs inside the drawerApp & Screens
Goal: keep the drawer tidy — one tab for fields, one for the rate table.
  1. Inside the drawer, add a Tabbed Container. Name the two tabs Treaty parameters and Rate table.
  2. On the Treaty parameters tab, add a Table showing the selected treaty's fields ({{ treatiesTable.selectedRow.extracted_json }}).
  3. On the Rate table tab, add a Table showing {{ treatiesTable.selectedRow.rate_json }}.
  4. At the top of the drawer, add a Text title showing the treaty ref and an Approve treaty button (you'll wire it in Part E).
Checkpoint: clicking a treaty shows its fields; switching to the second tab shows its rates.
Part E · owner: Review & Demo + Data & Database

8 · Part E — Edit, approve & export

Goal for this part: a reviewer can fix any value, the app checks it's valid, and Approve locks it in. Then you can export for the other systems.

Step E1 — Let reviewers edit the fieldsReview & Demo
Goal: a person can correct a wrong value on screen.
  1. Select the fields Table in the drawer. In the Inspector, open the value column's settings and turn on Editable.

What this does: Retool automatically remembers every change the reviewer makes — you don't need to save each cell. The whole set is saved when they Approve.

Checkpoint: you can type a new value into the table.
Step E2 — Check the data before approving (validation)Data & Database
Goal: stop obvious mistakes — blank required fields, wrong formats.
  1. Query panel → + New → Transformer. Name it validate. Paste the snippet below.
// returns ok:true only when the basics look right
const f = extractClean.value || {};
const problems = [];
if (!f.treaty_reference) problems.push("Missing treaty reference");
if (!f.effective_date)   problems.push("Missing effective date");
if (f.quota_share_pct != null &&
    (f.quota_share_pct < 0 || f.quota_share_pct > 100))
    problems.push("Quota share must be 0–100");
return { ok: problems.length === 0, problems };

What this does: a simple checklist the app runs for you. If anything's wrong, validate.value.problems lists it.

Checkpoint: validate.value.ok is true for a good treaty.
Step E3 — The Approve button (save & lock)Data & Database
Goal: Approve saves the checked values and marks the treaty Approved.
  1. Query panel → + New → Resource query → Retool Database → GUI mode → Action: Update a record → Table: treaties.
  2. Filter by: id equals {{ treatiesTable.selectedRow.id }} (this picks the right treaty).
  3. In the changeset set: status = approved, reviewed_json = {{ paramsTable.data }}, rate_json = {{ rateTable.data }}, reviewed_by = {{ current_user.email }}, reviewed_at = {{ new Date().toISOString() }}.
  4. Name it updateTreaty.
  5. Select your Approve treaty button → Event handlers → + Add → Event: Click → Action: Control query → Trigger updateTreaty.
  6. On the button, set Disabled to {{ !validate.value.ok }} so it only works when the data passes.
  7. On updateTreaty, add Event handlers for Success: trigger getTreaties (refresh the list), close reviewDrawer, and show a success notification.

What this does: saves the reviewer's checked values, flips the treaty to Approved, records who and when, then refreshes the list.

Checkpoint: click Approve — the drawer closes and the treaty now shows Approved in the list.
Step E4 — Export for the other systemsReview & Demo
Goal: get the data out as a file for iRate / the ATF.
  1. On the rate table (or fields table), open the table's ⋯ menu and use Export data → CSV.
  2. Or add a Button named "Export" and, in its Click event, choose Export data and pick the table.
Checkpoint: you get a CSV/Excel file with the treaty's data.

You now have a working app. Upload a treaty → AI fills it in → it's saved → a reviewer checks and approves → export. That's the whole story. Everything below is optional polish and demo prep.

Only if you have spare time

9 · UI polish (nice-to-have)

If the app works and you have time, these quick touches make it look finished — no custom code needed. Skip anything you're short on time for; a working plain app beats a pretty broken one.

Group things in Containers

Drag a Container around related components and give it a title (e.g. "Upload", "All treaties"). Instantly tidier.

Colour the status column

On the list table, set the status column type to Tag — Retool colours the values, so Approved/In Review stand out.

Right-align the numbers

In the rate table, set the number columns' alignment to right so the decimals line up and read cleanly.

Add a title bar

Put a Text component at the top: "Treaty Extractor". Big heading style. Gives the app an identity.

Friendly labels

Rename buttons and headings in plain words ("Approve treaty", "All treaties"). Use normal sentence case, not ALL CAPS.

Change the theme colour

Settings → Appearance lets you pick one accent colour that applies everywhere — far easier than styling each piece.

Loading & empty messages

Tables have built-in settings for "no rows yet" text and a loading spinner — turn them on so blank states don't look broken.

Add icons to buttons

Buttons have an icon setting — a checkmark on Approve, an upload arrow on the file button. A small, free touch.

Please avoid hand-written CSS or custom components during the hackathon — they eat time and break easily. Everything above is built into Retool's settings panels.

Plain-word dictionary

10 · Glossary

Quick, jargon-free meanings. Search to jump to one.

Treaty

The reinsurance contract your app reads. Signed between an insurer and a reinsurer (RGA).

ATF

The admin form/record where the treaty's details need to be entered. One place your data feeds into.

iRate

A pricing tool the rate tables get uploaded into. The other place your data feeds into.

Rate table

The grid of pricing numbers inside a treaty (rates by age, sex, smoker). Your app pulls this out too.

App

The screen you're building in Retool — buttons, tables, and boxes.

Component

A drag-on piece: a table, a button, a file upload, a text box.

Query

An action the app runs: read a file, ask the AI, save to the database.

Resource

A connection the app uses — the AI service, the database. Set up by your admin.

Retool Database

A built-in place to store data in tables. Where treaties are saved.

GUI mode

A way to build a database action by filling in boxes instead of writing code. You'll use this to save and update.

AI action / Generate text

A query that asks an AI model to do something — here, pull the fields out of the treaty.

Convert document to text

The built-in AI query that reads the words out of a PDF. If it's not available, use the Workflow + Python "Plan B" in Step A2.

Workflow

A separate mini-program in Retool that runs a few steps in order, away from your app screen. We use one in Plan B to read the PDF's text with a small script.

pypdf (Python)

A free tool for reading PDFs. The Plan B Workflow uses it to pull the text out of a treaty. You just paste our script — no coding needed.

Transformer

A small helper that tidies data so it displays neatly. You copy ours in.

Event handler

A "when this happens, do that" rule. Example: when the AI finishes, save the treaty.

On success

The kind of event handler that runs after a query finishes without an error.

Drawer

A panel that slides in from the side to show details — here, a treaty's fields and rates.

Tabs (Tabbed Container)

A component with switchable tabs. You use two: Treaty parameters and Rate table.

Segmented Control

A row of buttons for picking one option — here, In Review / Approved / All.

JSON

A tidy way to write data as labelled values. The AI returns its answer as JSON; you rarely read it directly.

Validation

Automatic checks that the data looks right before it's approved.

Status (In Review / Approved)

A treaty's stage. New treaties are "In Review"; once a person confirms, they become "Approved" and lock.

Mock data

Practice/fake data. The only kind you're allowed to use in the hackathon.
